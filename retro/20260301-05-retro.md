# Retrospective — 20260301-05

**Task:** epic 4.1 — many more phonetic variations in the generator
**Date:** 2026-03-01
**Team:** solo

---

## Summary

Expanded `frontend/lib/generator.ts` from 10 phonetic rules and 6 creative techniques to 49 rules and 15 techniques. `makePhoneticSwap` was replaced by `makePhoneticVariants` which emits one variant per matching rule instead of stopping at the first match. All 29 unit tests and 5 E2E tests passed first run.

## What went well

- The spec was precise and directly translatable to code — rule tables mapped 1:1 to array entries.
- Choosing to export `PHONETIC_RULES`, `CREATIVE_TECHNIQUES`, and `makePhoneticVariants` allowed clean unit tests without reaching into implementation details via `generateVariants` alone.
- The TDD cycle caught one test mistake early: "brrr" was chosen as a no-match word but `rr → r` fires on it; switching to "by" fixed it before implementation even started.
- All retro lessons applied correctly: `bun` prefixed with full path, orphan containers cleared with `--remove-orphans`, Playwright invoked with `--config`.

## What was hard

- Determining which test words genuinely match zero phonetic rules after adding single-vowel rules. Any word with a vowel now matches at least one rule; needed a vowel-free, consonant-cluster-free word ("by") for the fallback test.
- The `(?<!n)g(?!h)` lookbehind pattern required careful mental tracing to confirm it correctly excludes `ng` clusters while still firing on standalone `g`.

## What I'd improve

- The single-vowel rules (`a→ay`, `e→ee`, etc.) fire globally, turning "banana" into "baynaynaynay" (all `a`s). This is funny, but a future epic could add a first-occurrence-only mode as an option for more readable results.
- No `tsc --noEmit` type check step yet — the action item from Epic 1 remains open.
