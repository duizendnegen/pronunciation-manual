## Summary

Implemented Web Speech API audio playback in `frontend/app.ts`: per-card ▶ Play / ■ Stop buttons, a global ▶ Play all / ■ Stop button, auto-play on results, Escape key cancellation, voice selection, and graceful degradation. Added 9 E2E tests; all 26 tests (17 pre-existing + 9 new) passed on the first Docker build.

---

## What went well

- Spec was exhaustive (state machine, exact button labels, voice selection algorithm, degradation rule) — zero ambiguity, no mid-cycle clarifications needed.
- Retro lessons applied correctly: `--config e2e/playwright.config.ts` used from the start, port 3001 alignment already in place.
- `Object.defineProperty` in `addInitScript` was the right way to mock/unmock `window.speechSynthesis` — direct assignment fails on read-only browser properties; this approach is reliable across Chromium versions.
- Keeping all audio state module-level (rather than a class) was the minimal choice and worked cleanly.

## What was hard

- **SpeechSynthesis mockability**: `'speechSynthesis' in window` is always `true` in headless Chromium (the property exists even if non-functional). Switching the feature detection to `!!window.speechSynthesis` and using `Object.defineProperty({ value: undefined })` in the "unsupported" mock was the fix.
- **Test design for stable states**: The mock must NOT fire `onend` (so the sequence stays in a stable "speaking" state for assertions). Decided early to make the mock non-completing, which kept all state-check tests simple.
- **Auto-play timing**: `setResults` sets `playAllBtn` before calling `startSequence`, so `startSequence` can read the reference and update its text immediately. Ordering mattered.

## What I'd improve

- Consider extracting audio logic to `frontend/audio.ts` once it grows — currently ~100 lines of audio code lives in `app.ts`. At Epic 4 (LLM integration) the file will grow further.
- The `cancelSpeech` call inside `setLoading` operates on stale DOM refs after `replaceChildren` — harmless but slightly surprising. A comment or dedicated `clearAudioState()` (without UI update) might be cleaner.
